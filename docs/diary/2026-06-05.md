## Mehrere Meta-Chips pro Eintrag (Person / Ort / Tätigkeit)

**Entscheidung:** Option A — Delimiter-Konvention in bestehenden `text`-Spalten (kein Schema-Wechsel). Zentrale Hilfen in `lib/entryMeta.ts`: `splitMetaValues`, `joinMetaValues`, `atomicMetaValues`, `metaTagKey`. Detektoren, WGARM-EC und Embeddings nutzen atomare Werte; Specs `pattern-mirror.md` + `embeddings.md` angepasst.

**Ausgangslage (Analyse 2026-06-04):**

- UI zeigt bei `person` bereits mehrere Chips, wenn Werte mit `,` / `;` getrennt sind (`lib/entryMeta.ts` → `splitMetaValues`).
- DB und Eingabe: weiterhin **ein** String pro Spalte (`person`, `location`, `activity`).
- **Phase 1** (`detectTagFrequency`): zählt den **gesamten** Spaltenwert als ein Tag — `"Tom, Anna"` ≠ zwei Tags.
- **Phase 2** (WGARM-EC): eine Transaktion `tag:person:{ganzer string}` — keine Co-Occurrence zwischen einzelnen Personen.
- **Embeddings:** ein Klammer-Block pro Spalte (`lib/embedding.ts`).

**Optionen Datenmodell:**

| Option | Aufwand | Nutzen |
|--------|---------|--------|
| A. Delimiter-Konvention in bestehenden `text`-Spalten | gering | schnell, aber fragil |
| B. `text[]` auf `entries` (analog `categories`) | mittel | saubere Queries |
| C. Junction-Tabellen `entry_persons` etc. | hoch | beste Autocomplete / Dedup |

**Algorithmus — was sich ändert:**

1. **Normalisierung:** zentrale Hilfsfunktion → atomare `string[]` pro Meta-Typ; von UI, `detectTagFrequency`, `entryToTransaction`, `buildEmbedInput` gemeinsam nutzen.
2. **tag_frequency:** pro Eintrag jedes atomare Tag zählt; ein Eintrag kann mehrere Buckets füttern. Copy: „Tom“ heißt dann „Einträge, in denen Tom vorkommt“, nicht Kausalität.
3. **WGARM-EC:** ein Item pro atomarem Tag → Regeln wie `{Tom, Bibliothek} → valence:negative` möglich; mehr Items → ggf. strengere `minSupport` / `minLift`.
4. **Embeddings:** Spec anpassen — z. B. `[Tom][Anna][Bibliothek] Text` statt einem Block.
5. **Risiko:** Hintergrund-Namen auf vielen Einträgen → mehr schwache „immer wieder“-Insights; ggf. Max-Tags pro Typ oder bessere Scoring.

**Spec / Tests:**

- `docs/specs/pattern-mirror.md`: „atomarer Tag-Wert“, Verhalten bei Multi-Tag-Einträgen.
- `docs/specs/embeddings.md`: mehrere Klammern.
- Tests in `lib/__tests__/patternDetection.test.ts` für Split + Multi-Bucket.

**Mock-/Seed-Daten:**

- `scripts/seed-entries.mjs` nutzt **Szenario-Bündel** (Text + Meta + Valenz konsistent), keine zufällige Person auf beliebigem Text.
- Tom nur im Kontext „Lerngruppe / Bibliothek“; Orte wie „Bibliothek“ mit wiederkehrendem Lern-Stress für sinnvolle tag_frequency-Demos.

**Offen:**

- Primary vs. alle Tags in Mirror-Copy („wenn Tom dabei war“).
- Backfill bestehender Einträge bei Modellwechsel.
- Ob `body_state` jemals multi-value wird (aktuell: nein).

**Referenz:** Vollständige Analyse im Chat 2026-06-04 (Agent-Handoff); Code: `lib/patternDetection.ts`, `lib/wgarmEc.ts`, `lib/entryMeta.ts`.

## Mirror Reminder — Flow + Chip-Labels

**Entscheidung:** Reminder-Frage und Chips erst nach **Weiter** auf der Wenn-Dann-Zeile (nicht beim Ausfüllen). Zweites **Weiter** speichert Intention + optionalen Chip → Summary.

**Chip-Labels:** ~~`Heute` · `Ein paar Mal` · `Eine Weile` · `Lieber nicht`~~ → **`Heute` · `3 Tage` · `Diese Woche` · `Lieber nicht`** (siehe Abschnitt „WGARM + Reminder …" unten).

**Grundlage:** UX aus Mirror-Flow-Diskussion; Pirolli-Fading (3×), non-direktive Sprache.

**Offen:** Spec `pattern-mirror.md` Zeile „Reminder when fields filled" anpassen.

## Mirror — klarere Tag-Frequency-Texte + Achsen auf Eintragskarten

**Problem:** `tag_frequency` bei positiver/negativer Valence nannte nur Intensität + Zeitraum („Mir ist etwas aufgefallen. Das begleitet dich schon lange …“) — ohne Tag (z. B. Bibliothek) und ohne Richtung.

**Entscheidung:** Pattern-Text nennt jetzt explizit `{label}` und „meist positiv" / „meist schwer". `patternTextFromCandidate` speichert Intro + Summary + Closing (nicht nur den ersten Satz). Eintragskarten zeigen unten links zwei Mini-Slider (−/+ und ich/andere) mit farbigem Punkt — `EntryGridAxisMini`, default an wenn `grid_x` gesetzt.

## WGARM + Reminder + Signifikanter Moment — Korrekturen

**Entscheidung (Review 2026-06-05):**

1. **WGARM MIN_N = 20** — `MIN_WGARM_ENTRIES` in `lib/wgarmEc.ts`; unter 20 Einträgen feuert WGARM-EC nicht (Spec `wgarm-ec.md`).
2. **Fallback-Template entfernt** — generischer Text „Mir ist aufgefallen: {antecedent} …" produzierte unsinnige Kombinationen (z. B. „abends, unter der Woche … 100%"). Regeln ohne dediziertes Template werden verworfen.
3. **Reminder ohne Companion** — Chips: Heute / 3 Tage / Diese Woche; Abschluss passiv („Wenn-Dann gespeichert."); Intros neutral.
4. **Neuer Erkenntnistyp** — `significant_moment` in `insight-types.md` (Spec-only, Implementierung offen).

**Grundlage:** Mirror-Review — App darf sich nicht als aktiver Begleiter positionieren; WGARM braucht statistische Basis.

**Offen:** `significance_score`-Berechnung; Detektor `detectSignificantMoment`.
